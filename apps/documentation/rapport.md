# Project Report — Kirona

## 1. Project Overview

### Functional Scope

Kirona is a social platform centered around cinema and TV series. It allows users to:

- **Discover content**: browse and search movies and series, filtered by genre or type, with data sourced from the TMDB API.
- **Manage a personal watchlist**: add/remove content, track a status (watching / completed / dropped / planning), and monitor progress.
- **Rate content**: assign a score from 1 to 5.
- **Interact with a community**: send, accept, or decline friend requests.
- **Communicate in real time**: direct messaging between friends via WebSocket (Socket.IO).
- **Manage their account**: registration, login, logout, email verification, password reset, profile update.

The following features are defined in the database schema but not exposed in the UI:
- Watchparty (synchronized viewing between users)
- Notifications
- User statistics (`user_stats`)
- Custom lists (`lists`, `list_items`)
- Activity log (`user_activity_logs`)

### Global Stack

Monorepo **pnpm workspaces + Turborepo**, fully in TypeScript 5.9.

| Layer | Technologies |
|-------|-------------|
| Backend | Express 5, Socket.IO, Drizzle ORM, PostgreSQL 18, Zod |
| Frontend | React 19, Vite 7, TanStack Router, TanStack Query, Zustand, TailwindCSS 4, Radix UI (Shadcn) |
| Shared packages | Generated Axios SDK (Orval), isomorphic config, logger |
| Tooling | Biome (lint/format), Vitest 4, Docker Compose, Phase (secrets), Husky + lint-staged |

## 3. Architecture

### Overview

The project is structured as a monorepo with two main applications and four shared packages:

```
apps/
  api/            – REST API + WebSocket (Express 5 + Socket.IO)
  front/          – React 19 SPA
  documentation/  – Documentation artifacts

packages/
  api-sdk/        – Typed Axios SDK, auto-generated by Orval from the OpenAPI spec
  config/         – Isomorphic environment configuration (Node + browser)
  logger/         – Colorized isomorphic logger
  vitest-presets/ – Shared Vitest configuration
```

For details: `apps/documentation/architecture/README.md`.

### Backend — Clean Architecture

Each feature is an isolated module in `apps/api/src/modules/`, organized into three strictly separated layers:

- **Domain** — entities, repository interfaces, business errors. No dependency on infrastructure.
- **Application** — use cases (one class per business operation), DTOs. Depends only on domain interfaces.
- **Infrastructure** — Drizzle ORM implementations, Express routes, external adapters. Implements domain interfaces.

Dependency injection is manual, performed at bootstrap of each module. Routes are declared via TypeScript decorators (`@Get`, `@Post`, etc.) — a `DecoratorRouter` automatically generates the corresponding Express `Router` instances.

12 modules are registered and exposed: `auth`, `users`, `categories`, `contents`, `movies`, `series`, `watchlist`, `peoples`, `watchparties`, `friendships`, `conversations`, `messages`.

For details: `apps/documentation/developer-guide/api/architecture/`.

### Frontend — Feature Architecture

Routing is file-based via TanStack Router (tree generated in `routeTree.gen.ts`). Protected routes are grouped under the `_main.tsx` layout.

Each feature is encapsulated in a module under `src/features/` (UI, hooks, stores, services). API services follow a **dual-export pattern**: `query*Service` for React hooks (TanStack Query) and `*Service` for imperative calls outside components.

Global state is managed by three Zustand stores: `auth` (persisted in `sessionStorage`), `theme` (persisted in `localStorage`), and `typing` (ephemeral — real-time typing indicators).

### Real-time

Socket.IO with `/messages` namespace. The client is a singleton with lazy initialization (`getWebsocketServices()`). Handshake authentication uses the `accessToken` cookie (httpOnly). Events cover: joining a conversation, sending a message, typing indicator.

For details: `apps/documentation/spec.md` §5.

## 4. Technical Choices

### Reproducible Development Environment (Devenv + Direnv + Phase)

The development environment is fully declared in `devenv.nix` (Nix) and activated automatically by Direnv (`.envrc`) when entering the project directory. This ensures all team members use exactly the same system tool versions (Node 24, pnpm 10.28.2, phase-cli) without manual configuration.

Secrets (API keys, JWT secrets, DB credentials) are managed by **Phase**: `pnpm dev` calls `phase run` which injects variables into the process without storing them in the repository or sharing them by email. The production deployment exports Phase secrets directly into Docker Compose (`phase secrets export`).

For contributors without Phase access, the `pnpm dev:local` script allows launching the project from a local `.env` file (copied from `.env.example`).

### Monorepo with Turborepo

The Turborepo pipeline orchestrates tasks while respecting inter-package dependencies (build, type-check, tests, SDK generation). The build cache avoids recompiling unmodified packages.

### Auto-generated API SDK (Orval)

The API exposes an OpenAPI spec (automatically generated from Zod schemas via `@asteasolutions/zod-to-openapi`). Orval derives a typed Axios client in `packages/api-sdk`. This choice ensures type consistency between backend and frontend without manual duplication.

### Dual-token JWT Authentication

Access token (15 min) carried in memory (Zustand + `sessionStorage`), refresh token (7 days) in an `httpOnly` cookie. On each refresh, the old token is revoked in the database (`revoked_at`) and a new one is stored (rotation with invalidation). Logout also revokes the token server-side via the `refresh_tokens` table.

### Zod as Validation Layer and OpenAPI Source of Truth

Input validation (body, query, params) and OpenAPI documentation generation both rely on the same Zod schemas. This reduces drift between validation and documentation.

### Drizzle ORM with Versioned Migrations

14 successive migrations since February 2026. The main schema (`schema.ts`, 3000+ lines) is the source of truth for the database structure. Indexes are optimized for frequent queries (messages by conversation, user activity, notifications).

### Biome (lint + format)

Biome replaces ESLint + Prettier. Rules are strict: mandatory `const`, explicit `import type`, no `console` (except in the logger). Combined with Husky + lint-staged, each commit triggers lint + type-check + tests + build.

### Docker Compose for Deployment

- Dev: PostgreSQL only via `docker-compose.dev.yaml`.
- Prod: API + Frontend (nginx) via `docker-compose.prod.yaml` with health checks. Secrets are injected by Phase.
